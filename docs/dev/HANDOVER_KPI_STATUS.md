### ИСНА и взыскание — что писал Алихан (и ответ Олжаса)

**Суть сообщения Алихана:** как и планировалось, Олжас передал в отделы указание **заполнять данные о взыскании**, но выявилась **проблема в самой ИС ИСНА**: либо данные **не попадают в БД**, либо **нет нужных таблиц или полей**. **Олжас подтвердил** ситуацию и сообщил, что **начали работу по исправлению** и **будут держать в курсе**.

**Как это влияет на pipeline**

| Этап | Влияние |
|------|--------|
| **Импорт ИСНА → `CompletedInspection` / KPI 2** | Пока в БД КГД нет полноценных данных взысканий, **ETL не может построить полный факт KPI 2** «из сырья ИСНА» — нечего или не всё маппить. Это **блокер качества данных**, не ошибка кода движка. |
| **Движок `calc_collection`** | Логика **не меняется**: считает сумму по уже загруженным строкам. **Меняется только наполнение** после того, как ИСНА начнёт отдавать данные в БД (или после ручного ввода/правок). |
| **Эталон «КГД + ручная правка»** | По методологии Олжаса монитор опирается на КГД с **возможностью ручной правки** — **обходной путь**: ввод/исправление сумм и `is_accepted` оператором (или импорт из Excel ДГД), пока ИСНА чинят. |
| **Разработка и тесты** | Юнит-тесты движка на **фикстурах** не блокируются. **Интеграционная** сверка KPI 2 с **боевой** выгрузкой ИСНА — реалистична **после** появления данных в БД или через гибрид с ручными данными. |

## 3. KPI 2 — Взыскание (макс. 40 баллов)

| | |
|---|---|
| **Ожидается** | Только **принятые после перепроверки** суммы (Excel ДГД, столбец **AT** «взыскание принято»). В системе: **`is_accepted=True`**. Отдельно: в КГД **не суммировать** строки взыскания с методом **«доп. налоговая отчётность»** (`de_collection_method_id = 3` в `ri_collected_amount_record`) — зафиксировано в `data_sources.md`. |
| **Сделано** | `calc_collection`: фильтр `is_accepted=True`, УНА, период. |
| **Открыто** | **Олжас подтвердил:** доп. налоговые отчётности (**метод 3**) **не входят в KPI**. В **`CompletedInspection`** пока нет поля метода — факт KPI 2 должен приходить из ETL **уже без** сумм по `de_collection_method_id = 3` (или после заведения поля/разбивки сумм — фильтр в `calc_collection`). **Отдельно:** в самой ИСНА **неполные данные взысканий в БД** — идёт исправление на стороне КГД (**сообщение Алихана**, подтвердил Олжас); до исправления **pipeline KPI 2 из сырья ИСНА** ограничен — см. подраздел **«ИСНА и взыскание»** выше. |

## 4. KPI 3 — Среднее доначисление на проверку (макс. 10 баллов)

| | |
|---|---|
| **Ожидается** | Числитель = **та же сумма**, что KPI 1; знаменатель = **все** завершённые проверки УНА за период (**без** отдельного исключения ДФНО по форме с 2026, **без** фильтра `is_counted` в формуле — Олжас). Порог: единый по всем ДГД (среднее по прошлому году × 1.2). Особенность: **80–89% исполнения даёт 0 баллов**, не 5. |
| **Сделано** | `calc_avg_assessment` приведён к тому же набору строк, что KPI 1 (без `is_counted`, без `exclude form_type ДФНО`). `_kpi3_plan()` — среднее по прошлому году без этих исключений. |
| **Открыто** | ETL должен заполнять те же сущности согласованно с KPI 1. Ручной `is_counted` для «быстрого анализа» (Олжас) **не** участвует в KPI 3 в движке — не путать с KPI 4. |

### 8.1. Что уточнить Алихану (если «не понял посыл»)

**Посыл такой:** заказчик (**Олжас**) уже подтвердил **бизнес-формулу** KPI 6 (доля отменённых к доначислениям KPI 1). Споров о методологии ему задавать не нужно. Нам от **разработки/ИСНА** нужно другое:

1. **Из каких таблиц и полей** в вашей БД взять **сумму отмены** по ДГД и периоду отчёта (одна цифра или агрегат строк), чтобы положить в нашу сущность `AppealDecision.amount_cancelled`.

Коротко: **не «пересчитай KPI»**, а **«подскажи, как из ИСНА собрать числитель KPI 6 так же, как в мониторинге»**.

**Текст можно отправить Алихану почти дословно** (при необходимости сократить):

1. **Выгрузка оригинальных таблиц** с данными по апелляциям оказалась непростой: оригинал БД обслуживают другие подрядчики; DSRC отдаёт копию с **реплики**, куда поступают данные с оригинала. **Через Олжаса** (методолог и заказчик КГД) направлен запрос инженерам, поддерживающим оригинальную БД, по дальнейшей выгрузке в реплику и далее в команду.

Основа: **`ri_appeal_act_res_notif`**, FK `base_document_id` и `ri_appeal_id`. **Отдельно запрошено** у Олжаса уточнить у инженеров доступ к таблице с **незашифрованными** суммами.

## 11. Риски и технический долг (кратко)

- **ИСНА / KPI 2:** на стороне ИС **дыра по взысканиям в БД** (Алихан / Олжас) — **импорт из ИСНА неполный** до доработки ИС; обход: **КГД + ручная правка** или ожидание фикса.  
- **KPI 2:** метода взыскания в ORM нет — исключение метода 3 на стороне **ETL** или будущее поле + фильтр в engine (**суммы по методу 3 в KPI не входят** — Олжас).  
- **KPI 5:** коды инициатора в доке, в engine — старые строковые `case_type`.  
- **KPI 6:** модель `AppealDecision` заполнена не из полной цепочки ИСНА; период фильтрации апелляций по датам в engine — проверить при ETL (сейчас суммируются все `is_counted` по региону без привязки к `date_from`/`date_to` в `calc_cancelled` — **стоит перепроверить** при подключении реальных данных).  
- **`init_formulas --force`:** обновляет конфиги формул в БД; после смены методологии выполнять осознанно.
